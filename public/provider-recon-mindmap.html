3. Detailed Recon Logic — Report by Report

1A. PNR Match Report

What We Compare

  • Exact match: ixigo PNR = provider PNR = airline PNR
  • Extra PNR: Provider has PNR that ixigo doesn't recognize
  • Orphan PNR: ixigo has PNR but provider/airline says "no record"
  • Duplicate PNR: Same passenger, same flight, multiple PNRs

Data Sources

  • ixigo: Booking DB (PNR, flight details, pax info)
  • Provider: Settlement/booking file (daily dump or API)
  • Airline: GDS/NDC retrieve response (real-time or batch)

Actions on Mismatch

  • Extra PNR → Check if ixigo has a failed booking for same pax/date → if yes, map it
  • Extra PNR → If no ixigo record, flag as potential liability (provider may charge us)
  • Orphan PNR → Verify with airline directly → if cancelled, initiate refund from provider

1B. Fare Mismatch Report

Published Fare Mismatch

  • Compare: ixigo stored published fare vs airline's actual tariff
  • Includes: base fare, YQ (fuel surcharge), YR, airport taxes, PSF, UDF
  • If ixigo published fare < airline fare → margin leakage
  • If ixigo published fare > airline fare → overcharge risk

Offered Fare Mismatch

  • Compare: Fare offered to user on ixigo vs fare charged by provider
  • Includes: convenience fee, discount, promo breakdown
  • If provider charged more than offered → ixigo absorbs loss
  • Flag bookings where offered ≠ published beyond threshold (±₹50)

Frequency

  • Daily batch (T+1) for all bookings from previous day
  • Real-time alert for mismatches > ₹500

3A. Cancellation State Match

Three-Way State Comparison

  • All cancelled: ixigo = cancelled, provider = cancelled, airline = cancelled
  • ixigo cancelled, airline active: Revenue leak — ticket still valid, refund not due
  • Airline cancelled, ixigo active: User has invalid ticket — urgent CX action
  • Provider says cancelled, airline says active: Provider error — escalate

Data Sources

  • ixigo: Cancellation API logs + booking DB status
  • Provider: Cancellation confirmation + settlement file
  • Airline: PNR status retrieve (cancelled / confirmed / open)

What This Calculates

  • Ageing: Days since cancellation → refund still not received
  • Provider Recoverable: Amount owed by provider = (Expected Refund) − (Actual Refund Received)
  • If Source A ≠ Source B → Provider applied different penalty than fare rules state
  • If Source B ≠ Source C → Provider acknowledged refund but didn't credit it
  • Ageing > 15 days → Auto-escalate to provider account manager

3C. Split PNR & Ancillary Refund

Split PNR Storage

  • When airline splits a PNR (e.g., partial cancellation of multi-pax booking)
  • Store mapping: Original PNR → [Split PNR 1, Split PNR 2, ...]
  • Track refund per split PNR separately
  • Risk: Refund issued against original PNR but split PNR still active

Seat/Meal Refund Mismatch

  • Compare: CMS refund condition (e.g., "refundable before 4 hrs") vs actual refund amount
  • Per-airline ancillary refund rules differ significantly
  • CMS says "full refund" but provider refunded ₹0 → escalate
  • CMS says "non-refundable" but user was charged refund penalty → overcharge

4. Tech Losses & Customer Delight → DWH

Tech Loss Categories

  • Booking failure loss: Price increased between user's search and booking attempt
  • API timeout loss: Provider API timed out, rebooked at higher fare
  • Duplicate booking loss: System created 2 PNRs, one needs cancellation
  • Fare feed error: Stale/incorrect fare cached → booked at loss
  • Ancillary booking failure: Seat/meal not booked but user was charged

Customer Delight Categories

  • Voucher issued: Compensation for poor experience
  • Fare difference absorbed: ixigo ate the loss to keep customer happy
  • Free upgrade: Seat/meal upgrade at ixigo's cost
  • Refund over-credited: Refunded more than entitled as goodwill

DWH Requirements

  • Push all tech loss + delight records to Data Warehouse daily
  • Dimensions: provider, airline, route, booking type, failure reason
  • Enable trending: is tech loss increasing for a specific provider?
  • Source owner: Shubham Gadia — get current logic + approval workflow

5. Reschedule Recon — Full Breakdown

Reschedule State Match

  • All aligned: ixigo = rescheduled, provider = rescheduled, airline = new flight
  • ixigo shows rescheduled, airline shows original: Reschedule didn't actually happen
  • Airline rescheduled, ixigo shows original: User has wrong itinerary — urgent

PNR / Split PNR Storage

  • Reschedule may issue a new PNR → map Old PNR → New PNR
  • Multi-pax reschedule (only some pax reschedule) → split PNR
  • Store: original PNR, new PNR, split PNRs, pax mapping per PNR

Reschedule Amount Mismatch

  • Fare difference: New fare − old fare (could be positive or negative)
  • Airline reschedule charges: Penalty/fee charged by airline
  • Compare: What ixigo charged user vs what airline actually charged
  • If airline charged ₹2,500 but ixigo charged user ₹3,000 → overcharge
  • If airline charged ₹3,000 but ixigo charged user ₹2,500 → ixigo loss

5. Stakeholders & Ownership

Primary Stakeholders

Supply / Partnerships Finance Product (Recon)
  • Supply Team: Owns provider relationship, resolves extra PNRs, fare disputes, SLA enforcement
  • Finance Team: Owns P&L impact — margin leakage, recoverable amounts, ageing, write-offs
  • Product (Recon): Builds the reports, defines match logic, exception handling rules

Secondary Stakeholders

  • Tech: Owns API integrations, event pipelines, DWH ingestion
  • Data: Builds dashboards, trend analysis, anomaly detection on recon data

6. Key Metrics & SLAs

Booking Recon Health

  • PNR Match Rate — % of bookings where all 3 sources agree (target: >99.5%)
  • Extra PNR Count — Monthly count of unmatched provider PNRs (target: 0)
  • Fare Mismatch Rate — % of bookings with >₹50 fare delta (target: <0.5%)
  • Avg Fare Delta — Average ₹ mismatch per booking (trend down)

Cancellation Recon Health

  • Cancel State Match Rate — % where all 3 sources agree (target: >99%)
  • Refund Match Rate — % where 3-way refund amounts match (target: >95%)
  • Avg Refund Ageing — Days from cancel to refund received (target: <7 days)
  • Provider Recoverable — Total ₹ owed by providers (trend down monthly)

Reschedule & Losses

  • Reschedule State Match — % aligned across sources (target: >99%)
  • Reschedule Amount Match — % within ±₹100 tolerance (target: >95%)
  • Monthly Tech Loss ₹ — Total loss attributed to tech issues (trend down)
  • Customer Delight Spend — Monthly ₹ on goodwill (track vs budget)
