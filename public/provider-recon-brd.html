1. Executive Summary & Objectives

1.1 Background

ixigo Flights works with multiple suppliers (airlines directly + aggregators like TBO, Cleartrip B2B, Riya, etc.) to book flight tickets. After a booking/cancellation/reschedule, ixigo's records must match the supplier's records and the airline's records. Discrepancies lead to revenue leakage, incorrect refunds, orphan PNRs, and poor customer experience.

Currently, reconciliation is largely manual or partially automated with gaps in coverage. This project builds a comprehensive, automated reconciliation system that matches ixigo data against provider/airline data across all lifecycle events (booking, cancellation, reschedule).

1.2 Business Objectives

  1. Eliminate revenue leakage — Detect fare mismatches, extra PNRs, and refund shortfalls that cost ixigo money
  2. Reduce provider recoverable ageing — Track and recover pending refunds faster (target: <7 days avg)
  3. Achieve 99.5%+ PNR match rate — Every booking on ixigo has a verified matching record at the provider and airline
  4. Automate exception handling — Auto-close small deltas (≤₹50), auto-escalate large ones (>₹500)
  5. Enable data-driven provider negotiations — Historical mismatch data as leverage for better SLAs and commercials
  6. Map all tech losses and customer delight to DWH — Enable trend analysis and root cause attribution

3. Functional Requirements — Booking Reconciliation

BR-001
PNR Match Report
The system shall compare PNR records across three sources (ixigo Booking DB, Provider settlement/booking file, Airline GDS/NDC retrieve) and produce a daily match report classifying each PNR as: Matched, Extra PNR (exists in provider but not ixigo), or Orphan PNR (exists in ixigo but not provider/airline).
Acceptance Criteria:
AC1: For every booking made on day D, the recon report for D+1 includes a match/mismatch status.
AC2: Extra PNRs are flagged with provider name, PNR, passenger details, and flight date.
AC3: Orphan PNRs trigger an automated airline PNR retrieve to confirm actual status.
AC4: Duplicate PNR detection (same pax + same flight + multiple PNRs) is highlighted separately.
AC5: Report is filterable by provider, airline, date range, and match status.
BR-002
Seat/Meal Details Match
The system shall compare ancillary (seat + meal) details stored in ixigo's Ancillary DB against the airline's PNR retrieve response (SSR codes). Any mismatch where a user paid for a seat/meal but it doesn't exist on the airline PNR must be flagged.
Acceptance Criteria:
AC1: Each booking with ancillary add-ons is checked against airline SSR confirmation.
AC2: Mismatches are categorized: seat mismatch, meal mismatch, both, or ancillary missing entirely.
AC3: For "ancillary missing" cases, an automated refund eligibility check is triggered.
AC4: Report shows: Booking ID, PNR, booked seat/meal, confirmed seat/meal, mismatch type.
BR-003
Published Fare Mismatch
The system shall compare the published fare stored in ixigo's booking record against the airline's actual tariff for that flight/class/date. The comparison must include component-level breakdown: base fare, YQ, YR, airport taxes (PSF, UDF), and any other surcharges.
Acceptance Criteria:
AC1: Every booking's published fare is compared at the component level (not just total).
AC2: Mismatches exceeding ±₹50 are flagged for investigation.
AC3: The report shows both the total delta and per-component deltas.
AC4: Historical trend of published fare mismatches per provider is available.
AC5: Margin leakage (ixigo published < airline actual) is calculated in ₹ per booking.
BR-004
Offered Fare Mismatch
The system shall compare the fare offered to the user on ixigo's search/booking flow against the fare actually charged by the provider. This captures the "what user saw" vs "what provider billed" gap, including convenience fees, discounts, and promo codes.
Acceptance Criteria:
AC1: Offered fare is captured from the search/booking log (pre-payment snapshot).
AC2: Provider charge is captured from the booking confirmation / settlement file.
AC3: Delta is calculated: offered − provider charge. Positive = ixigo profit, Negative = ixigo loss.
AC4: Promo code / discount bookings are tagged separately (offered fare may intentionally differ).
AC5: Report supports drill-down by provider, airline, route, and fare class.

4. Functional Requirements — Cancellation Reconciliation

CR-001
Cancelled State Match
The system shall perform a 3-way state comparison for every cancellation: ixigo cancellation status vs provider cancellation status vs airline PNR status. Any state disagreement is a critical mismatch requiring immediate action.
Acceptance Criteria:
AC1: All cancellations from day D are checked for state consistency by D+1.
AC2: State combinations are classified with severity:
  - All cancelled → Match (auto-close)
  - ixigo cancelled + airline active → P0 — Revenue leak
  - Airline cancelled + ixigo active → P0 — User has invalid ticket
  - Provider cancelled + airline active → P1 — Provider error
AC3: P0 mismatches trigger automated alerts to Supply and CX teams within 1 hour.
AC4: Report is filterable by provider, airline, cancellation date, and severity.
CR-002
Refund Amount 3-Way Match
For every cancellation, the system shall perform a 3-way refund comparison:
Source A: ixigo's assumed penalty based on B2B fare rules / API response
Source B: Provider's penalty as stated in their cancellation/settlement file
Source C: Actual refund amount credited to ixigo's account by the provider
Acceptance Criteria:
AC1: All three sources are compared for every cancellation. Tolerance: ±₹10.
AC2: Mismatch between A and B → "Penalty Mismatch" — provider applied different rules than expected.
AC3: Mismatch between B and C → "Credit Mismatch" — provider acknowledged refund but didn't pay it.
AC4: The system calculates Provider Recoverable = Expected Refund (A) − Actual Received (C).
AC5: Ageing is tracked: days since cancellation date where Provider Recoverable > ₹0.
AC6: Auto-escalation rules: Ageing >7 days → alert to Supply; >15 days → alert to Supply + Finance lead; >30 days → flag for write-off review.
AC7: Monthly provider-wise recoverable summary is generated for Finance.
CR-003
Split PNR Storage (from Airline)
When an airline splits a PNR (e.g., partial cancellation in a multi-passenger booking), the system shall store the mapping between the original PNR and all resulting split PNRs. This mapping is critical for accurate per-passenger refund tracking.
Acceptance Criteria:
AC1: A "PNR Split Map" table stores: Original PNR → [Split PNR 1, Split PNR 2, ...] with passenger mapping.
AC2: Split PNRs are detected via provider notification, airline PNR retrieve, or manual entry.
AC3: Refund tracking is per-split-PNR (not just per original PNR).
AC4: The cancellation recon report links split PNRs back to the original booking.
AC5: Alert when a refund is issued against the original PNR but split PNRs are still active.
CR-004
Seat/Meal Refund Mismatch
The system shall compare the ancillary refund amount against the CMS-defined refund conditions for that specific airline and ancillary type. Different airlines have different refund policies for seats and meals (some are fully refundable, some have time-based conditions, some are non-refundable).
Acceptance Criteria:
AC1: For every cancelled booking with ancillaries, the system checks CMS refund rules for that airline.
AC2: CMS condition (e.g., "refundable if cancelled >4 hrs before departure") is evaluated against the actual cancellation timestamp.
AC3: Expected ancillary refund (per CMS) is compared with actual ancillary refund (from provider).
AC4: Mismatch report shows: CMS rule, expected amount, actual amount, delta, airline, provider.
AC5: Supports ancillary types: seat, meal, excess baggage, priority boarding.

5. Functional Requirements — Reschedule Reconciliation

RS-001
Reschedule State Match
The system shall compare reschedule status across ixigo, provider, and airline for every reschedule transaction. A "state mismatch" means the user's itinerary on ixigo doesn't match what the airline actually has — leading to the user showing up for the wrong flight.
Acceptance Criteria:
AC1: All reschedules from day D are checked by D+1.
AC2: State combinations classified:
  - All rescheduled → Match
  - ixigo rescheduled + airline shows original → P0 — Reschedule didn't happen
  - Airline rescheduled + ixigo shows original → P0 — User has wrong itinerary
AC3: P0 mismatches trigger automated CX alert to proactively inform the user.
AC4: Airline-initiated schedule changes (not user-initiated) are tracked separately.
RS-002
Rescheduled PNR / Split PNR Storage
When a reschedule results in a new PNR (common with some airlines/aggregators), the system shall store the old PNR → new PNR mapping. For multi-passenger bookings where only some passengers reschedule, the split PNR scenario must also be handled.
Acceptance Criteria:
AC1: "PNR Reschedule Map" stores: Old PNR → New PNR, with flight details for both.
AC2: Multi-pax split reschedules store: Old PNR → [New PNR (rescheduled pax), Old PNR (remaining pax)].
AC3: Passenger-level mapping is maintained (which pax on which PNR).
AC4: ixigo's Trip Details page reflects the new PNR for rescheduled passengers.
RS-003
Reschedule Amount Mismatch
The system shall compare the amount charged to the user by ixigo against the actual amount charged by the airline. Reschedule amounts have three components: fare difference (new fare − old fare), airline reschedule charges (penalty/fee), and any applicable taxes.
Acceptance Criteria:
AC1: For every reschedule, the system captures and compares:
  - Fare difference: (new flight fare) − (old flight fare)
  - Airline reschedule charge/penalty
  - Taxes on the above
AC2: Total charged to user by ixigo vs total charged by airline/provider — delta calculated.
AC3: Overcharge (ixigo charged more than airline) flagged for potential user refund.
AC4: Undercharge (ixigo charged less than airline) flagged as ixigo loss.
AC5: Tolerance: ±₹50. Beyond tolerance → investigation ticket created.
AC6: Report drill-down by provider, airline, route, and reschedule reason (user vs airline-initiated).

6. Functional Requirements — Change Logs & Audit Trail

CL-001
State Change Tracking
The system shall log every state change in a booking's lifecycle as an immutable event with timestamps. This creates a complete timeline from booking creation through confirmation, modification, cancellation, reschedule, and refund.
Acceptance Criteria:
AC1: Every state transition is logged with: booking ID, old state, new state, timestamp (IST), triggered by (system/user/agent/airline/provider).
AC2: Timestamps are captured from both ixigo side and provider side (when available).
AC3: The delta between ixigo timestamp and provider timestamp is calculated and stored.
AC4: Supports querying: "Show me all state changes for booking X in chronological order."
AC5: Log is append-only (immutable) — no updates or deletes allowed.
CL-002
Timestamp-Based Reporting
The system shall generate reports based on timestamp analysis to identify delays, SLA breaches, and process bottlenecks. For example: time from cancellation request to provider acknowledgment, time from provider acknowledgment to refund credit.
Acceptance Criteria:
AC1: Report shows duration between key milestones:
  - Booking: search → payment → PNR confirmation
  - Cancellation: user request → ixigo processing → provider cancel confirm → refund credit
  - Reschedule: user request → ixigo processing → provider reschedule confirm → new PNR
AC2: SLA breach detection: flag transactions where any step exceeds defined thresholds.
AC3: Provider-wise SLA performance dashboard.
CL-003
Before/After Delta Logging
For every modification event (fare change, seat change, PNR change, reschedule), the system shall store the before and after values to enable precise impact analysis.
Acceptance Criteria:
AC1: Change log format: { field, old_value, new_value, changed_at, changed_by }.
AC2: Fields tracked: PNR, flight number, departure date/time, fare components, seat, meal, passenger details.
AC3: Linkable to specific recon mismatches (e.g., "this fare mismatch was caused by this change event").

7. Functional Requirements — Tech Losses & Customer Delight

TL-001
Tech Loss Identification & Categorization
The system shall identify all financial losses attributable to technical issues (not user or airline actions) and categorize them by root cause. Tech losses are ixigo-absorbed costs that should be tracked for engineering prioritization and provider negotiation.
Acceptance Criteria:
AC1: Tech loss categories include (at minimum): API timeout, fare feed error, duplicate booking, ancillary booking failure, payment-booking state mismatch, rebook at higher fare.
AC2: Each tech loss record includes: booking ID, loss amount (₹), category, root cause description, provider, airline, route.
AC3: Automated classification where possible (e.g., if rebooked within 5 min at higher fare → "fare feed error").
AC4: Manual override for classification (Supply/Finance can re-categorize).
TL-002
Customer Delight Mapping
The system shall track all customer delight actions (vouchers, fare absorption, upgrades, over-refunds) and map them back to the originating booking event. This enables Finance to track goodwill spend and Product to measure its impact on retention.
Acceptance Criteria:
AC1: Delight records include: booking ID, delight type, amount (₹), approved by, approval date, originating event (booking failure, wrong seat, delayed refund, etc.).
AC2: Monthly delight spend report by category, provider, and originating event type.
AC3: Linkage to tech loss where applicable (e.g., delight issued because of a tech loss).
TL-003
DWH Integration
All tech loss and customer delight records shall be pushed to the Data Warehouse for cross-functional analytics, trending, and dashboarding.
Acceptance Criteria:
AC1: Daily ETL pushes tech loss + delight data to DWH.
AC2: DWH dimensions: provider, airline, route, booking type, loss category, delight type, time period.
AC3: Dashboard showing: monthly tech loss trend, top loss categories, top providers by loss, delight spend vs budget.
AC4: Data source and current logic to be obtained from Shubham Gadia.

Dependency: Tech loss categorization logic and customer delight approval workflow are currently owned by Shubham Gadia. A handoff meeting is needed to document the existing logic before building the automated version.

12. Risks & Dependencies

Risk / DependencyImpactLikelihoodMitigation
Provider file format inconsistency — Each provider sends data in different formats High High Build provider-specific parsers with a common output schema. Start with top 3 providers by volume.
Provider file delays — Some providers send T+2 or T+3 instead of T+1 Medium High Design the system to handle late-arriving data. Re-run recon when late files arrive. Track provider SLAs on file delivery.
Airline GDS access limitations — Not all PNRs can be retrieved in bulk High Medium Prioritize provider file as primary source. Use airline retrieve only for mismatches and spot checks.
CMS fare rules not up-to-date — Stale rules lead to incorrect expected penalties Medium Medium Build a CMS staleness check. Alert Supply when fare rules haven't been updated in >30 days for active routes.
Tech loss categorization logic undocumented Medium High Schedule handoff meeting with Shubham Gadia. Document current logic before building automated version.
Split PNR detection relies on provider notification Medium Medium Supplement with periodic airline PNR retrieve for bookings with partial cancellations/reschedules.
Engineering bandwidth — Multiple integrations with different providers High Medium Phase rollout by provider priority (highest volume first). Reusable parser framework.

14. Glossary

TermDefinition
PNRPassenger Name Record — unique booking identifier assigned by airline
Split PNRWhen an airline divides one PNR into multiple PNRs (e.g., partial cancellation in a group booking)
Published FareThe airline's official tariff for a flight/class/date, filed with regulatory bodies
Offered FareThe fare shown to the user on ixigo's platform (may differ from published fare due to markup/discount)
ProviderThe entity through which ixigo books the ticket — could be the airline directly or an aggregator (TBO, Cleartrip B2B, Riya, etc.)
Settlement FileA periodic file sent by the provider listing all bookings, cancellations, and financial transactions
SSRSpecial Service Request — airline industry code for ancillaries (seats, meals, wheelchair, etc.)
CMSContent Management System — ixigo's internal system storing fare rules, refund policies, and ancillary conditions per airline
B2B Fare RulesThe fare/cancellation/reschedule rules applicable to ixigo's B2B relationship with the provider (may differ from retail rules)
Provider RecoverableThe amount owed by the provider to ixigo — typically pending refunds from cancellations
AgeingNumber of days a refund/recoverable amount has been outstanding since the originating event
GDSGlobal Distribution System — backend systems (Amadeus, Sabre, Travelport) used for airline inventory and booking
NDCNew Distribution Capability — IATA's modern API standard for airline retailing
DWHData Warehouse — centralized data store for analytics and reporting
Tech LossFinancial loss caused by a technical issue in ixigo's systems (not user/airline fault), absorbed by ixigo
Customer DelightGoodwill gestures (vouchers, refunds, upgrades) given to customers as compensation for poor experience
T+1One day after the transaction date — the standard turnaround for daily reconciliation
RACIResponsibility matrix — Responsible, Accountable, Consulted, Informed
